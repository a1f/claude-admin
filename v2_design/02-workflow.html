End-to-end workflow

Three stages: Plan → Breakdown → Dispatch & complete. Each PR loops through coder → reviewers → critics → architect decision → CI → user approval → merge → self-improve. A milestone is "done" when its final critique passes.

Architect always explains what the merge does and why it's safe before asking — never just "ready to merge?".

After merge: self-improvement & plan reconciliation

  • · Self-improvement proposal. Architect compares (a) original PR spec vs (b) what shipped vs (c) reviewer findings. Files a tiny meta-plan: "next time, prompt coder to do X earlier" / "reviewer missed Y, add to checklist".
  • · Plan reconciliation. Architect re-reads the plan with the merged PR in mind. If new info changes a downstream PR, update its spec. If a milestone goal needs adjustment, surface to user.
  • · Activity log. Every decision (dispatch, drop, iterate, accept, merge) logged with rationale. Replayable on the plan timeline.

Final milestone critique

When the last PR in a milestone merges, a heavyweight critic runs against the milestone goal — not any single PR. It re-reads the milestone description, inspects the merged code, tests the deployed feature end-to-end, and answers: did we actually deliver what we said we would?

Output

  • · Verdict: passed / gaps found
  • · Per-goal trace: each milestone goal → which PR(s) addressed it → evidence
  • · Gap list: things the milestone said it would do but didn't — becomes new PRs in the same milestone, looping back into dispatch
Brainstorm prompts.
  • Should the architect auto-merge after user approval, or always require human-clicks-the-button on GitHub?
  • What's the threshold for "iterate vs drop"? Hard rules or per-project config?
  • Critique scoring: 1-100 numeric, or rubric-based (5 axes × pass/fail)?
  • Should self-improvement proposals be auto-applied (to claude_admin's own prompts), or always human-reviewed?
