ca_v1 — final plan

Local-machine Rust orchestrator: a daemon that owns claude sessions, an architector per (repo, milestone), task-processors per dispatched task, and a TUI client. v2_design's web admin grows from this foundation later (web server replaces TUI as another client; remote-machine support added).

§2.5

tmux session topology

Every claude invocation runs inside a tmux pane managed by the daemon — no orphaned background subprocesses. Reason: live attachability (tmux attach -t claude-admin shows what every agent is doing right now) without giving up the structured-JSON output the orchestrator depends on.

ConcernChoice
SessionSingle shared session named claude-admin. Daemon creates it on startup if missing.
Window per architectorWindow named <repo>:<milestone> with one pane: the long-running architector claude session.
Window per dispatched taskWindow named <task-id> opened on dispatch. Holds the coder pane plus the reviewer/critic panes spawned at review time.
Pane runtimeCoder = interactive claude. Everyone else = claude -p --output-format stream-json --include-partial-messages | tee <state-dir>/log.jsonl so the pane scrolls AND the JSON stream lands on disk for the daemon to parse.
Output capturetee inside the pane (clean JSON, no terminal escapes). tmux pipe-pane reserved for the interactive coder pane where stdout isn't a clean stream.
Pane lifecyclePane auto-closes when the agent process exits (default tmux behavior — no remain-on-exit). Logs survive on disk under ~/.work/dispatches/<plan>/<task-id>/; daemon reads exit code from the JSON stream tail, not the pane.
CleanupDaemon reaper sweeps dead windows + orphan panes on every accept loop tick. ca task abort sends SIGTERM to the pane's foreground process, then closes the pane.
Reusable managerOne tmux::SessionManager in ca-daemon built in M2. M4 reuses it for sub-action planning panes; M5 for reviewer + critic fan-out; M9 for milestone-closure critique.

§4

User control

  • Settings declare which decisions are automated vs require user.
  • Always require user input when: dramatic plan change, blocker can't be resolved, architector confidence-low, fix-round 3 reached.
  • User-input requests show in TUI top-of-screen alert; user replies via ca user-respond.
§5

Validation discipline

  • Every accepted task must produce evidence: tests run, screenshots, e2e scenarios, demo recordings.
  • Architector cannot accept until required validators pass with stored artifacts.
  • Artifacts visible in TUI; auditable post-merge.

What's NOT in v1

  • Web admin (that's the next plan: v2_design)
  • Remote machine bridges (ca_v1 is single-machine; protocol is designed remote-ready)
  • Multi-user (single-user assumption)
  • Plugin marketplace, mobile UI, complex permission models
  • Auto-merge after CI green (user clicks ✓ on merge for v1; auto-merge can come later)
