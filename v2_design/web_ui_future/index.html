claude_admin v2 — design brief

Goal: replace the current Rust TUI/CLI/daemon stack with a self-hosted web admin tool that orchestrates Claude (and Codex) work across projects — from planning, through breakdown, dispatch, review, critique, validation, all the way to merge — with a laptop-side daemon that connects out to the central server.

Start here → 00 · Final plan — the canonical, locked spec. All decisions, the renumbered M0a–M13 roadmap, and the next-step pointer to defining workflow skills.
How to use this brief. Open each page below in your browser. The diagrams are interactive (Mermaid). Pages 01–05 are the architecture + working roadmap. Pages 06–11 are clickable UI mocks of the target screens — they use fake data and illustrate flows. Iterate by editing files in v2_design/.

Open questions to settle next

  • Backend stack: Rust+axum (familiarity) vs TypeScript end-to-end (frontend reuse) vs Go (simplicity). See 01 · stack candidates.
  • DB: Postgres (managed, multi-process safe) vs SQLite (single-binary, simpler ops). Likely Postgres given remote daemon.
  • Auth: GitHub OAuth (single-user, good audit trail) vs magic-link email. GitHub OAuth recommended since we already integrate with gh.
  • Frontend: React+Vite vs SvelteKit vs HTMX+server-rendered. We can decide after the mocks settle.
  • Agent runtime: claude CLI in tmux (current) vs Claude Agent SDK (programmatic) vs hybrid. Big decision — see 04 · agents.
  • Notion: keep as a sync target for plans, or fully replace? Mocks assume full replace but with markdown export.
