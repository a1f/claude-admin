Milestone roadmap v1 — iterating

Twenty-two milestones. The big shift from v0: API milestones and UI milestones are now separate. Each backend milestone ships an API testable end-to-end via curl + integration tests. UI milestones plug into existing APIs. A dedicated UI polish milestone lives late, after dogfooding has surfaced the rough edges. Foundation is split into server / daemon / docker-e2e. Sessions and Plans expanded.

What changed since v0 (your feedback)

  • M0 split into 3: M0a server skeleton · M0b daemon skeleton · M0c docker e2e infra (containers + integration tests).
  • M1 split into 3: M1a sessions list API only · M1b sessions content + state · M1c dashboard UI.
  • Plans expanded: M2a plans CRUD API · M2b plan comments API (Google-doc-style threaded comments anchored to sections / ranges) · M2c history/versioning API · M2d plans editor UI.
  • Breakdowns split: M3a breakdown API + GH issue mirror · M3b breakdown UI.
  • Repo/GitHub manager added: M4a server-side primitives for worktree + branch + PR ops via daemon, before dispatch builds on them. M4b dispatch API. M4c dispatch UI.
  • Review loop split: M5 reviewers backend · M6a critics + architect backend · M6b PR review UI (the big screen).
  • Validation split: M7a runners + artifacts API · M7b artifact gallery UI · M8 CI + final approval (UI is light, bundled).
  • UI polish promoted to its own milestone: M10. Empty/loading/error states, keyboard shortcuts, density, a11y, perf — after we've dogfooded for a while.
  • Codex moved to M11 (was M10).

The API/UI separation pattern

Every feature now goes through three kinds of milestones. Sequencing is generally infra → API → UI, with a polish pass at the end.

infraFoundation / plumbing. Sets up environments and primitives nobody sees: server, daemon, docker compose, repo manager. Validated with integration tests.
apiAPI milestones. Add features as REST endpoints + DB schema. Tested via curl, integration tests, e2e against containers. UI is irrelevant here. Each shippable on its own — you can use the API directly while we wait for the UI.
uiUI build milestones. Plug into one or more existing APIs. The screen ships when the API is already proven. No backend work allowed in a UI milestone.
polishUI polish. Single dedicated milestone late. Empty states, loading skeletons, error toasts, keyboard shortcuts, density, a11y, animations.

Milestones

M0ainfra · foundation

Server skeleton + auth + deploy

Web server runs on a real VPS at https://your-server. GitHub OAuth login works. /healthz green. Reproducible deploy.
  1. Repo scaffold (chosen stack), formatter / linter / test runner / CI
  2. HTTP server skeleton with /healthz
  3. GitHub OAuth flow + session cookie + protected routes
  4. Deploy pipeline: Caddy + systemd + Postgres + migration runner, single command
  5. Smoke tests for OAuth + auth-required endpoint
PRs: 5Risk: low–medExit: kill server, redeploy in <5min, login still works
M0binfra · foundation

Daemon binary + WSS pairing

Run ca-daemon pair <token> on your laptop, see "1 daemon online" in the browser. Survives kill -9 / wifi drop / reconnect.
  1. Daemon binary + persistent config in ~/.config/claude-admin/
  2. Pairing protocol: token-based, server-issued, single-use
  3. WSS reverse-tunnel + heartbeat + exponential backoff reconnect
  4. Daemon registry on server: online/offline/last-seen, stub UI table
PRs: 4Risk: medium (network edge cases)Exit: daemon survives chaos test (random kill + restart loop)
M0cinfra · foundation

Docker e2e infra

CI spins up server-and-daemon-in-containers, runs integration tests covering pairing + heartbeat + WS RPC. Every PR runs them.
  1. Server Dockerfile (multi-stage, slim runtime)
  2. Daemon Dockerfile (Linux, with tmux pre-installed for tests)
  3. Docker compose harness for tests + ephemeral Postgres
  4. First e2e suite: pair → heartbeat → daemon RPC echo round-trip
  5. CI integration: matrix runs on PR, results posted as check
PRs: 5Risk: mediumExit: a green PR reliably means "deploy is safe"
M1aapi · observe

Sessions discovery API (list-only)

Daemon discovers Claude sessions in tmux. Server stores session rows. GET /sessions returns a list per project. No state inference, no pane content yet.
  1. Tmux discovery in daemon (port heuristic from current ca-lib)
  2. Sessions schema + REST CRUD (create from daemon push, read from API)
  3. Daemon → server push protocol for session create/destroy
  4. Integration test: spawn fake tmux session in daemon container → assert it appears in API
PRs: 4Risk: lowExit: curl returns the list of running sessions
M1bapi · observe

Sessions content + state

API exposes session state (idle/working/needs_input/done) + most recent ANSI pane snapshot. WS subscription pushes live updates. Snapshots TTL'd.
  1. State inference (idle / working / needs_input / done) from pane content + activity
  2. Pane snapshot capture (debounced, ANSI-preserved) on daemon
  3. Snapshot storage: append-only ring buffer + TTL
  4. WS subscription endpoint for live session updates
  5. e2e test: state changes propagate from daemon to subscribed WS client
PRs: 5Risk: low–medExit: WS client sees state flip working→needs_input within 2s
M1cui · observe UI

Sessions dashboard UI first real screen

First user-facing screen. Lists projects + sessions with state badges + pane preview. Replaces the TUI in your daily flow.
  1. UI shell (sidebar + topbar + auth flow + project switcher)
  2. Sessions list page with state badges + filter
  3. Pane preview component (xterm.js or equivalent for ANSI)
  4. Session detail page with full pane + event feed (events stub if not yet built)
PRs: 4Risk: lowExit: you stop opening tmux to check on Claude
M2aapi · author

Plans API (CRUD)

Plan + Milestone schema + REST. No UI. Tested via curl + integration tests.
  1. Plan schema (body_md, goal, motivation, validation_strategy, status)
  2. Milestone schema + plan→milestone relationship
  3. REST CRUD for plans + milestones
  4. e2e test: create plan, add 3 milestones, query, update, delete
PRs: 4Risk: lowExit: "create a plan from curl" works end-to-end
M2bapi · author

Plan comments API Google-doc-style

Threaded comments anchored to a plan section, milestone, or text range. Open / resolve / reply / @mention. No UI.
  1. Comment schema + anchor model (kind: section / milestone / range / breakdown_pr)
  2. Threaded replies (parent_id) + status (open / resolved)
  3. REST endpoints: create / list-by-anchor / reply / resolve
  4. @mention parsing (deferred notifications until M10 polish)
  5. e2e test: create comment on a range, reply, resolve, list-by-plan returns thread tree
PRs: 5Risk: medium (anchor model has to survive plan edits)Exit: resolved comments stay attached even when surrounding plan text moves
M2capi · author

Plan history / versioning API

Every save creates a new immutable version row. Diff endpoint, revert endpoint. No UI.
  1. plan_versions table: full body snapshot per save
  2. Diff endpoint: structured diff between any two versions
  3. Revert endpoint: non-destructive (creates new version matching old)
PRs: 3Risk: lowExit: revert v3 → v1 produces v4 identical to v1
M2dui · author UI

Plans editor UI

Author plans in the browser, replacing your Notion flow. Plugs into M2a/b/c APIs.
  1. Editor + live preview (CodeMirror + remark)
  2. Embedded mermaid renderer in preview
  3. Image upload (drag-drop, paste) → object store
  4. Milestone list inline with drag-reorder
  5. Comments rail: threaded comments anchored to selection / sections
  6. History view: version timeline + diff + revert button
PRs: 6Risk: medium (comments anchoring UX is the hard part)Exit: next plan you write is in claude_admin, not Notion
M3aapi · author

Breakdown API + GH issue mirror

Per-milestone breakdown of PlannedPRs with parallelization. Mirrored bidirectionally to a GH issue.
  1. Breakdown + PlannedPR schema (spec_md, acceptance_criteria_md, depends_on, status)
  2. REST CRUD for breakdowns + planned_prs
  3. GH issue mirror: write breakdown body to GH issue, watch for issue updates
  4. e2e test: create breakdown → GH issue exists with content; edit either side stays in sync
PRs: 4Risk: medium (sync edge cases)Exit: a breakdown round-trips through a GH issue without diverging
M3bui · author UI

Breakdown UI

Visual editor for a milestone's PlannedPRs: DAG of dependencies, status kanban-ish list, link out to GH issue.
  1. Breakdown page shell + list of PlannedPRs
  2. DAG / dependency editor (drag edges or "depends on" select)
  3. Status view: filter by planned/coding/reviewing/merged
  4. Embed in plan view as a sub-tab per milestone
PRs: 4Risk: medium (DAG editor UX)Exit: click a milestone → break it down in the UI → GH issue auto-populated
M4aapi · dispatch

Repo + worktree + GitHub manager primitives

Server-side primitives: list branches, create/remove worktree on a daemon, list PRs, open draft PR — all via daemon over WSS. No dispatch yet, just the building blocks.
  1. Worktree manager on daemon (git worktree add/remove + tracked state)
  2. Branch ops API (list, create, delete) routed to daemon
  3. PR ops via gh on daemon (create-draft, view, comment)
  4. Server-side manager API that fans out to the right daemon
  5. Integration tests in docker e2e: server creates worktree on daemon container, opens draft PR against fake GH
PRs: 5Risk: medium (gh + worktree lifecycle is fiddly)Exit: POST /worktree → branch + dir on disk; tear down cleanly
M4bapi · dispatch

Dispatch API + coder spawn

Server picks a daemon for a PlannedPR, daemon creates worktree (M4a), spawns Claude in tmux with assembled prompt. PlannedPR transitions planned → dispatched → coding → drafted.
  1. Dispatch endpoint + daemon picker (label-based)
  2. Prompt assembly: spec_md + plan_context + PR rules → markdown blob
  3. Spawn cmd in daemon: tmux new-window + send prompt + start Claude
  4. GH PR detection loop: when draft PR opens, link to PlannedPR + transition state
PRs: 4Risk: high (prompt quality determines coder output)Exit: POST /dispatch results in a draft PR with no terminal use
M4cui · dispatch UI

Dispatch UI

"Dispatch" button on breakdown rows; live linkage in dashboard ("Coder for M3.PR2 of project X"); dispatch history per PlannedPR.
  1. Dispatch button + confirm modal (which daemon? attempts so far?)
  2. Dashboard linkage: session row shows the PlannedPR it's coding
  3. Dispatch history view per PlannedPR (attempts, outcomes)
PRs: 3Risk: lowExit: click → coder writes & pushes a draft PR with no terminal use
M5api · review

Reviewers

When a PlannedPR enters "drafted", auto-run reviewers via Claude API. Findings stored, exposed via API. No UI yet.
  1. Job queue (Postgres-backed)
  2. Reviewer worker + Claude API client + structured output parsing
  3. Prompt templates per kind: security / bugs / quality / custom
  4. Findings storage + REST endpoints
PRs: 4Risk: high (prompt quality)Exit: on a real PR, at least 2 of 3 reviewer kinds produce useful findings
M6aapi · gate

Critics + architect decision

Critics score 1–100 across axes. Architect aggregates reviewer + critic output → drop / iterate / accept with rationale. Iterate sends a feedback bundle to the coder. No UI yet.
  1. Critic worker + adversarial prompt templates per kind
  2. Architect decision logic (rules + LLM rationale)
  3. Feedback bundle generator (curated markdown)
  4. Iterate flow: coder receives bundle, re-pushes, reviewers + critics re-run
  5. Decision audit log + replayable inputs
PRs: 5Risk: high (this is the brain)Exit: for one real PR, the iterate loop works without your terminal intervention
M6bui · gate UI

PR review UI the big screen

The most-viewed page (mock 10). Reviewer findings, critic score card, validation gallery placeholder, architect verdict, decision UI.
  1. PR review page shell + tabs (overview / reviewers / critics / validation / diff / timeline)
  2. Reviewer findings tab (grouped, filterable, severity badges)
  3. Critic score card with axes breakdown
  4. Decision UI: accept / iterate / drop with override + rationale display
  5. Timeline tab: full event stream
  6. Files / diff tab (link out for huge diffs)
PRs: 6Risk: medium (UX-heavy)Exit: you read findings + decide in claude_admin instead of GH
M7aapi · gate

Validation gate (runners + artifacts API)

Mandatory validators per project. Playwright + e2e scripts + migration dry-run produce Artifacts. Approval gate refuses approval until required validators have passed.
  1. ValidationRun + Artifact schema + runner abstraction
  2. Playwright runner on daemon (daemon container has chromium for tests)
  3. Artifact storage + signed URLs (local FS first)
  4. E2E script runner (shell exec on daemon, capture stdout)
  5. Approval gate enforcement at API level
PRs: 5Risk: medium (Playwright on real dev envs)Exit: a UI PR cannot reach approval without ≥1 screenshot
M7bui · gate UI

Validation gallery UI

PR review screen shows artifacts inline. Click-to-zoom, video playback, diff/log viewers. Validation status panel.
  1. Gallery component (thumbnails + click-to-zoom modal)
  2. Viewer modals for video / log / trace
  3. Validation status panel: which validators required, ran, passed/failed
PRs: 3Risk: lowExit: screenshots open in-page, zoomable
M8api+light ui · merge

CI watcher + final-approval flow

Watch GH check_suite events. Fix-on-red loop. Architect drafts "ready to merge" explanation. User approves → claude_admin merges via gh. UI is light here — bundled.
  1. GH webhook receiver (with polling fallback)
  2. Fix-on-red loop: parse failure logs, send to coder via feedback bundle
  3. "Ready to merge" composer (architect drafts changes / risks / rollback)
  4. User-approval UI in PR review screen
  5. Merge automation: gh pr merge --squash; PlannedPR → merged
PRs: 5Risk: low–medExit: first end-to-end merge with no terminal actions
M9api+ui · merge

Self-improvement + final milestone critique

Per-merge retros. Per-milestone heavyweight critique reads merged code + tests + plan, verdict pass / gaps_found. Gaps become new PlannedPRs. Milestone shipped UI.
  1. Self-improvement worker: post-merge retro markdown
  2. Retrospective view per project (apply / dismiss)
  3. Final critique runner (Opus): post-last-merge, full milestone evaluation
  4. Gap → PlannedPR loop: gaps create new PRs, milestone state in_flight again
  5. Milestone shipped celebratory UI with timeline
PRs: 5Risk: medium (final critique prompt quality)Exit: first milestone shipped end-to-end through claude_admin
M10ui · polish UI POLISH

UI polish pass after dogfooding

After ~weeks of dogfooding through M9, dedicated milestone to fix the rough edges that real use surfaces. No new features.
  1. Empty / loading / error / offline state audit across every screen
  2. Keyboard shortcuts pass (⌘K, gd/gp/gs, ⌘↵ to approve, etc.) + cheatsheet modal
  3. Density toggle + theme polish (spacing, font sizing, color contrast)
  4. Accessibility audit: tab order, ARIA labels, screen-reader pass on key flows
  5. Performance pass: pane snapshot streaming, large plan rendering, virtualized lists
PRs: 5Risk: lowExit: claude_admin feels like a shipped product, not an MVP
M11infra · polish

Codex + multi-laptop

Add Codex as a runtime alongside Claude. Pin projects to specific daemons. Multi-laptop dispatch.
  1. Codex session detection in daemon (process + pane heuristic)
  2. Runtime field on Session + UI runtime badges
  3. Daemon labels + project pinning ("this project always goes to lap-A")
  4. Multi-daemon dispatcher: pick laptop based on labels + availability
PRs: 4Risk: lowExit: Codex shows alongside Claude with no special handling

What this roadmap deliberately doesn't include

  • Multi-user / teams
  • Mobile UI · iOS app
  • Plugin/skill marketplace
  • Live multi-tab plan editing (last-write-wins for v1)
  • Notion sync (mocks assume claude_admin replaces Notion)
  • Customizable dashboards / saved views
  • Cost forecasting (we track actuals only)

Open knobs to turn together

  • Is M2 (3 API milestones + 1 UI) the right shape? Could collapse M2c history into M2a if you'd rather get the editor sooner.
  • Should M6b PR review UI come before M6a critics? You'd see findings + your own brain decides until critics catch up. Useful early dogfooding.
  • M4a (repo manager) — should this come earlier (after M0c) so it's available for everything that needs git ops, or stay where it is (right before dispatch)?
  • UI polish at M10 vs splitting it into 2–3 smaller polish milestones interleaved (e.g., one after M1c, one after M2d, one at M10)?
  • Is "M8 has light bundled UI" cheating the API/UI split? Could make it M8a + M8b. Tradeoff: M8 UI is genuinely tiny (one approval modal).
  • Plan comments (M2b) — anchor only to plan body, or also to milestones / breakdown PRs / sessions / decisions? More anchor types = more value but bigger scope.
