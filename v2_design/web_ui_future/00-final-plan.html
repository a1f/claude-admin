Workflow summary

Three nested loops: PR loop inside milestone loop inside feature loop. Full sequence + state machines in 02 · workflow.

  1. Plan — write the plan in claude_admin (motivation, design, validation strategy, milestones with descriptions). Mermaid + images embedded. Versioned.
  2. Breakdown — pick a milestone, break into PlannedPRs (~150 LOC each). Mirror to a real GitHub issue.
  3. Dispatch — architect picks parallelizable PRs; daemon creates worktrees; spawns coder.
  4. Coder drafts a real GH PR. Daemon detects, transitions PlannedPR → drafted.
  5. Reviewers run (security / bugs / quality) — find defects in the code.
  6. Critics run (goal_alignment / acceptance_check / plan_drift) — score 1-100, find defects in goal-fit.
  7. Architect aggregates → drop / iterate / accept with written rationale. Iterate sends feedback bundle to coder.
  8. Validation gate — Playwright + e2e + migration dry-runs. Mandatory artifacts before approval.
  9. CI watcher — promote draft → ready, fix-on-red loop with coder.
  10. User approval — architect drafts "ready to merge" explanation; you click ✓.
  11. Merge — claude_admin runs gh pr merge --squash.
  12. Self-improvement — post-merge retro per PR; final critique per milestone with gap detection.
§4

Agent runtime: all-CLI

All agents run via the claude CLI to stay covered by your subscription quota. No per-token API cost.

RoleWhere it runsInvocation
coderLaptop daemonLong-running interactive claude in a tmux pane, in a fresh worktree. Reads spec_md + plan_context as initial prompt.
reviewersServerclaude -p "<reviewer prompt + diff>" subprocess; capture stdout; parse JSON-formatted findings.
criticsServerSame pattern as reviewers; adversarial prompts; JSON output with score + axes.
architectServerDecision logic is rule-based (rules + thresholds in code). When LLM rationale needed: claude -p subprocess.
Final critiqueServerHeavy claude -p run, reads merged code + tests + plan + acceptance criteria.
Quota safety. All-CLI means subscription quota is the single point of failure. Heavy month: 16M tokens (coder-dominated) + 5M tokens (reviewers/critics). If your tier saturates, claude_admin work stops. Mitigations baked into M0a + M6:
  • · Daemon reports monthly token usage to server every hour.
  • · Server pauses dispatch + shows banner at 80% of quota.
  • · Hard-stop at 100%: no new agent runs, current ones complete.
  • · Per-coder-attempt time/token budget; over-budget halts that attempt.
  • · "Re-run reviewer" requires explicit user click; never automatic.
