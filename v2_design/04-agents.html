Agents & orchestration

Five roles. The architect is the boss; everyone else is a tool the architect uses. Coder is interactive (long-running tmux session, can ask questions). Reviewers, critics, validators are headless — short Claude API calls returning structured output.

Roles at a glance

architect  Architect

Server-side. Owns dispatch & decisions. Never touches code itself.

Inputs

  • · Plan + breakdown for context
  • · Reviewer findings + critic scores per PR
  • · CI signals, validation artifacts
  • · User overrides

Outputs

  • · Dispatch decisions (which PRs to start in parallel)
  • · Drop / iterate / accept verdicts with rationale
  • · Coder feedback bundles
  • · User-approval requests with explanation
  • · Self-improvement & reconciliation notes

coder  Coder

Long-running interactive Claude in a tmux pane on a laptop, working in a fresh worktree.

Inputs

  • · PlannedPR.spec_md + acceptance_criteria_md
  • · Plan context (parent goal, what other PRs do)
  • · Repo at base branch + worktree
  • · On iterate: feedback bundle from architect

Outputs

  • · Pushed branch + draft GH PR (linked back to PlannedPR)
  • · "ready" signal (via daemon hook) — all tasks done, tests written, PR drafted
  • · Notes / open questions on entries it deferred

Limits

  • · Cannot merge. Cannot delete branches. Cannot rewrite history of other PRs.
  • · Sandboxed to its own worktree path.
  • · Time / token budget per attempt; over budget → architect notified.

reviewer  Reviewer

Headless Claude API call. Multiple kinds run in parallel against the diff.

Variants

  • · security — auth, secrets, injection, supply chain
  • · bugs — logic errors, edge cases, off-by-one
  • · quality — style, naming, dead code, abstraction
  • · perf (when flagged) — hot path, allocation, query shape
  • · custom (per-project) — user-defined checklists

Output contract

Structured JSON: {summary, findings: [{severity, file, lines, desc, suggested_fix}]}. See 03 · data model.

critic  Critic

Headless Claude API call with strong adversarial prompt. Scores 1–100.

Variants

  • · goal_alignment — does the diff actually do what the PlannedPR said?
  • · acceptance_check — does each acceptance criterion have evidence?
  • · plan_drift — does this PR break assumptions in downstream PRs?

Why separate from reviewer

Reviewers chase defects in the code. Critics chase defects in the goal-fit. A PR can be defect-free and still solve the wrong problem. Critics catch that.

Score floor

Default thresholds: ≥80 accept, 60–79 iterate, <60 drop. Per-project tunable.

CI watcher

Polls/listens to GitHub checks for the PR. Reports green/red back to architect. On red, bundles failing logs and forwards to the coder for fix-on-red.

Validator

Runs the validation gate: playwright e2e, screenshot capture, migration dry-run, perf bench, etc. Stores results as Artifacts. Can run on the laptop (via daemon) or on the server.

Iterate feedback bundle

When the architect chooses iterate, it doesn't dump raw reviewer JSON at the coder. It produces a curated bundle: here's what to fix, in what order, why. 

// example feedback bundle (markdown sent to coder)
## Iteration #2 — please address the following before re-drafting

### Blockers (must fix)
1. **`src/auth/session.ts:82-109`** — session token in localStorage. The `security` reviewer
   flagged this; we agreed at plan time we'd use cookies. Switch to httpOnly cookies, fall back
   to localStorage only when SameSite=None is unavailable.

2. **Acceptance criterion AC-3 not satisfied** — "logged-in user can refresh and stay logged in".
   The current implementation doesn't persist auth across reload because the cookie isn't set on
   `/auth/callback`. Add the cookie before the redirect.

### Nits (optional, fix if cheap)
- `src/ui/Sidebar.tsx` — magic number 240. Extract `SIDEBAR_WIDTH_PX`.

### Notes from critic (score 71/100)
The critic flagged plan drift: this PR added a `User.preferences` field that's not in the data
model in the plan. Either remove it (preferred) or update the plan in milestone M2.

### What we are NOT changing
The cookie-vs-localStorage decision is final — don't ask again.

Architect prompt template (sketch)

The architect itself is an LLM call when it needs to write rationale or feedback. The decision logic is rule-based + LLM rationale.

You are the Architect for project {project.name}. Plan: {plan.title}.
Milestone {milestone.title} goal: {milestone.goal}.

PlannedPR {pr.id}: {pr.title}
Spec:
{pr.spec_md}

Acceptance criteria:
{pr.acceptance_criteria_md}

Diff (truncated to {N} files): {diff}

Reviewer findings:
{reviews_json}

Critic verdicts:
{critiques_json}

Validation:
{validation_summary}

Decide: drop / iterate / accept.

Output JSON:
{
  "decision": "drop" | "iterate" | "accept",
  "reasoning_md": "...",
  "feedback_bundle_md": "..." (if iterate),
  "drop_reason_md": "..." (if drop),
  "next_step_for_user": "..." (if accept)
}

Self-improvement & final critique

Self-improvement (after each merge)

Compare what we said the PR would do vs what shipped vs what reviewers caught. Output is a tiny markdown file added to a per-project retrospective.md. User reviews; some get applied to prompt templates.

Final critique (after milestone)

Heavy. Re-reads the milestone description, walks merged code, runs the test suite end-to-end, exercises deployed feature via Playwright. Verdict: passed / gaps_found. Gaps become new PlannedPRs in the same milestone — milestone state goes back to in_flight.

Brainstorm prompts.
  • How do we keep coders honest about "tests written" — a final reviewer check that asserts test coverage delta?
  • Should architect have an explicit "ask user mid-PR" channel, or is going-back-to-plan the only escape hatch?
  • For critique adversarial framing: do we use a single critic with multiple axes, or N independent critics that never see each other?
  • Should the daemon stream coder progress (e.g. partial commits) so architect can intervene mid-run, or wait for coder to signal "ready"?
