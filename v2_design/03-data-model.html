Data model

The shape of "the work". Every entity here exists because some screen needs to show it, some agent needs to decide on it, or some flow needs to recover from a crash on it. If an entity isn't pulling its weight on at least one of those axes, we drop it.

Identifier strategy

  • · UUIDv7 for all primary keys. Time-ordered, monotonic, debug-friendly. Makes pagination trivial.
  • · Slugs on Project for URLs (/p/claude_admin/...) — humans-readable.
  • · Foreign keys enforced at the DB layer. No "soft" relationships in jsonb.

Soft delete vs hard delete

EntityStrategyWhy
ProjectSoft (archived_at)Want to browse old plans / past PRs even after we stop the project.
Plan, Milestone, PlannedPR, Decision, Critique, Review, ValidationRunHard delete forbiddenAudit trail. We replay history to learn.
CommentSoft delete (status=deleted)Resolved comments stay queryable; deleted ones hide but persist for thread integrity.
Session, PaneSnapshotTTL hard delete (90 days default)Volume. Keep events; lose pane bytes.
WorktreeMark removed_atSo we know if a session is orphaned vs current.
Artifact blobsTTL after milestone shipped (configurable)Storage hygiene. Metadata stays forever.

Indexes & access patterns

Hot reads

  • · "All sessions for daemon X live right now" → (daemon_id, state) index on session.
  • · "Newest pane snapshot for session" → (session_id, seq DESC) on pane_snapshot.
  • · "All PRs in milestone, in order, with statuses" → (breakdown_id, order_idx).
  • · "Activity feed for project" → event(session_id) + session→project join, OR materialized view v_activity.

Brainstorm prompts.
  • Do we want event sourcing for the architect decisions, or normal CRUD with append-only Decision rows is enough? Recommend the latter for v1.
  • Should PlannedPR.depends_on be a hard constraint (cannot dispatch until merged) or advisory (architect considers it)?
