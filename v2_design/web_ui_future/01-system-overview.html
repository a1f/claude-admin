System overview

Three tiers: laptop(s) running the daemon and Claude/Codex sessions in tmux, a self-hosted web server that holds the database and orchestration logic, and a browser UI that's the only thing you actually look at. The daemon connects out, so your laptop can be behind any NAT.

Daemons dial out — no inbound ports on your laptop. Server has a single public TLS endpoint.

Components

ca-daemon (laptop)

  • · Discovers Claude/Codex sessions in tmux (process + pane content heuristics)
  • · Streams pane snapshots (debounced, ANSI-preserved) to server
  • · Reports session state: idleworkingneeds_inputdone
  • · Executes server-issued commands: spawn session, send keys, git worktree add, run validator
  • · Maintains heartbeat over WSS, auto-reconnect with backoff
  • · Local-only fallback: if server unreachable, queues events to disk and replays

Web server (self-hosted VPS)

  • · HTTP API: projects, plans, breakdowns, PRs, sessions, artifacts
  • · WebSocket gateway: bidirectional with daemons, push-only with browser
  • · Orchestrator: phase scheduling, agent dispatch, decision logic
  • · Job queue (bg workers): reviewer/critic runs, CI watchers, validators
  • · Postgres: durable state. Object store (S3-compat or filesystem) for blobs
  • · Auth: GitHub OAuth (single-user initially, multi-user later)

Browser UI

  • · Single SPA — see UI mocks 06–11
  • · Pure consumer of HTTP API + read-only WebSocket subscriptions
  • · No business logic; orchestrator owns decisions, UI just visualizes
  • · Deep-links: every plan / breakdown / PR / session has a stable URL

Agent runtimes (sandboxed)

  • · coder Long-running interactive Claude in a tmux pane on a laptop, working in a git worktree. Drafts the actual PR.
  • · reviewer Headless Claude API call given the PR diff + plan context. Returns structured findings.
  • · critic Headless Claude API call given the PR diff + the goal it should achieve. Returns score + rationale.
  • · architect Server-side. Combines reviewer + critic output → decision (drop / iterate / accept).

Why this shape

PropertyWhy it falls out of the topology
Laptop is offline-tolerantDaemon queues events on disk if WS drops. No work lost when wifi flakes.
Server is the single source of truthPlans, decisions, history all on the server. Daemon is a thin executor; you can blow it away and re-pair.
Multiple laptops just workEach pairs with the same server. UI shows all sessions across both. Nothing extra to configure.
NAT-friendlyDaemon dials out only. No port forwarding, no Tailscale required.
Easy to add Codex (or others)Daemon just needs a session-detector for the new tool. Server treats them all uniformly via session.runtime field.

Reverse-tunnel WebSocket protocol (sketch)

Both directions over one persistent connection. Daemon authenticates with a long-lived pairing token; server multiplexes RPCs to a specific daemon by daemon_id.

{"v":1,"type":"session.update","daemon_id":"lap-A","ts":"...","payload":{
  "session_id":"...","state":"working","tmux":{"session":"main","window":"3","pane":"0"},
  "preview_hash":"sha256:..."
}}

{"v":1,"type":"pane.snapshot","daemon_id":"lap-A","payload":{

Failure modes & degraded behavior

FailureBehavior
Daemon WS dropsDaemon queues events to local SQLite. UI shows daemon as offline. On reconnect, replays queue ordered.
Server downUI shows banner. Daemon retries with exponential backoff. Sessions keep running locally; you lose live tracking, not work.
Postgres downServer returns 503. UI shows banner. No data corruption (single source).
Claude API rate-limit during reviewer/criticJob queue requeues with backoff. Architect waits, doesn't block other PRs.
Coder gets stuck (idle >N minutes mid-task)Architect notifies user. Optional auto-nudge with "still there?" — see 04 · agents.
Worktree on disk goes missingDaemon detects on next op, marks session orphan, server flags PR for human review.
